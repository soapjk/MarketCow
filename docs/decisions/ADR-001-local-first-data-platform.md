# ADR-001：本地优先、分层存储与统一 API

状态：建议采纳

日期：2026-07-15

## 背景

多个项目分别抓取行情和财务数据，会重复处理代码映射、限流、缓存、字段变化和质量问题。全市场漏斗还要求批量扫描、公告时间和历史可复现性，单个机器人内的 SQLite API 已不足以长期承担这一职责。

## 决策

- 新建独立 `marketcow`，成为本地市场数据的唯一推荐入口。
- 原始、标准化、派生三层分离。
- 以 Parquet + DuckDB 作为首期分析存储，以本地 HTTP API 和维护 CLI 对外提供契约。
- A 股底座采用通达信/Mootdx/BaoStock 的批量路线，AKShare 用于深度补齐，官方披露用于核验。
- 允许从既有本地应用只读导入目录和历史数据，随后逐 provider 迁移。
- 首期仅面向本机/内网可信环境，不对公网发布或提供上游原始数据下载。

## 后果

正面影响：

- 下游不再关心每个免费接口的细节。
- 能够统一做缓存、回退、质量审计和 point-in-time 查询。
- 全市场筛选与单公司深度研究使用同一份事实底座。

成本与约束：

- 需要维护稳定 ID、标准字段和 schema migration。
- 免费来源变化仍不可避免，只是被隔离在 provider 层。
- 数据源条款与社区许可证需要持续登记。
- 本地单写架构不适合立即扩展成公网高并发服务；这是首期有意接受的限制。

## 未选择的方案

- 继续把抓取逻辑堆在机器人中：耦合继续扩大，其他项目仍需重复实现。
- 仅依赖 AKShare：接口丰富，但网页上游变化和字段口径风险过于集中。
- 直接 fork SimTradeData：功能接近，但 AGPL 与本项目未来服务形态需要额外许可判断，且字段仍需大量扩展。
- 首期直接上 PostgreSQL/Kafka/对象存储：当前本地规模没有足够收益，显著增加运维负担。
